# 周备有谱：开发模型与提示词手册（Cursor 可直接复用）

## 使用说明（统一口径）

- 适用项目文件：`weekly-prep-docs/01-product/product-prototype-mobile.html`、`weekly-prep-docs/01-product/product-prototype.html`、`weekly-prep-docs/02-architecture/bugFix.md`、`weekly-prep-docs/02-architecture/tech-architecture.md`、`weekly-prep-docs/03-process/plan.md`、`weekly-prep-docs/01-product/demo.md`、`weekly-prep-docs/03-process/开发计划与验收SOP.md`
- 当前策略默认模型：`Kimi K2.5`（成本更友好）
- 代码主力模型：`GPT-5.3 Codex`
- 复杂问题升级：`GPT-5.4` -> `Opus 4.6`
- 多文件联动执行：`Composer 1.5`
- 说明：是否“真正可用”以你本地 `PASS-模型名-时间` 测试结果为准

### 快速选择规则

- 低成本优先：`Kimi K2.5` / `Grok Code` / `GPT-5.3 Codex Low Fast`
- 质量平衡：`Sonnet 4.6` / `GPT-5.3 Codex`
- 高难任务：`GPT-5.4` / `Opus 4.6`

## Prompt 1：移动端原型优化（UI/UX）

**推荐模型**
- 首选：`GPT-5.3 Codex`
- 低成本备选：`Kimi K2.5`
- 升档：`Sonnet 4.6`（当涉及交互重构或复杂样式冲突时）

**可直接复制**

```text
你是资深前端与移动端体验设计工程师。请基于当前项目中的 `weekly-prep-docs/01-product/product-prototype-mobile.html` 优化页面可用性与视觉层级，必须直接修改代码，不只给建议。

目标：
1) 提升信息层级清晰度（标题、价格、按钮、状态）
2) 提升操作效率（点击热区、底部导航、关键路径）
3) 提升可读性（对比度、字号、间距、留白）

约束：
- 不改变核心业务流程与信息架构
- 不引入新框架，保持原生 HTML/CSS/JS
- 优先复用现有 class 命名，避免大规模破坏性重构

输出要求：
- 直接给出可运行的完整改动（按文件输出）
- 明确列出你改了哪些模块（如：商品卡片、购物车浮层、结算区、底部导航）
- 每项改动补一句“为什么这样改”

验收标准：
- 关键按钮可点击区域 >= 44px
- 颜色对比可读（重要文字和背景有明显区分）
- 首屏关键 CTA 清晰可见
- 不出现布局错位和滚动穿透问题
```

## Prompt 2：BugFix 清单转可执行修复计划

**推荐模型**
- 首选：`GPT-5.3 Codex`
- 低成本备选：`Kimi K2.5`
- 疑难偶现问题：`GPT-5.4` 或 `Opus 4.6`

**可直接复制**

```text
请读取并分析 `weekly-prep-docs/02-architecture/bugFix.md`，将问题清单重构为“可执行修复计划”，并给出可落地的修复顺序。

请按每个 bug 输出以下结构：
- 严重级别（P0/P1/P2/P3）
- 影响范围（页面/模块/用户路径）
- 稳定复现步骤（最少 3 步）
- 根因假设（技术层面）
- 最小修复方案（尽量小改动）
- 回归测试点（至少 3 条）
- 上线风险与回滚建议

额外要求：
- 优先处理影响支付、下单、地址、库存一致性的问题
- 把可并行处理和必须串行处理的任务分开
- 输出一个 1 周内可完成的修复排期（按天）
```

## Prompt 3：技术架构文档增强（可落地版）

**推荐模型**
- 首选：`Opus 4.6`
- 平衡备选：`GPT-5.4`
- 文档整理收敛：`Sonnet 4.6`

**可直接复制**

```text
请审阅 `weekly-prep-docs/02-architecture/tech-architecture.md`，将其升级为“可落地执行”的架构文档。重点从以下维度补强：
1) 可维护性（模块边界、职责划分、命名规范）
2) 扩展性（活动高峰、品类扩展、多城市）
3) 稳定性（容错、重试、降级、幂等）
4) 可观测性（日志、指标、告警、追踪）
5) 成本与效率（开发成本、运维成本、迭代效率）

输出格式要求：
- 先给“现状问题 Top 10”
- 再给“分阶段方案 P0/P1/P2”（每项含收益、风险、工作量）
- 给出关键接口或模块的建议目录结构
- 给出最小可行改造路径（先改什么、后改什么）

约束：
- 不做空泛理论，必须结合买菜业务场景
- 每条建议都能执行、可验收
```

## Prompt 4：开发计划排期重构（带 DoD）

**推荐模型**
- 首选：`Kimi K2.5`
- 平衡备选：`Sonnet 4.6`
- 复杂依赖分析：`GPT-5.4` 或 `Opus 4.6`

**可直接复制**

```text
请基于 `weekly-prep-docs/03-process/plan.md` 输出一版“可交付、可验收、可跟踪”的迭代计划，按周拆解。

每条任务必须包含：
- 输入（依赖条件）
- 输出（交付物）
- 负责人角色（前端/后端/测试/产品）
- 完成定义 DoD（Definition of Done）
- 风险与缓冲（含最坏情况）

额外要求：
- 标注关键路径任务（Critical Path）
- 标注可并行任务与阻塞关系
- 给出每周里程碑与验收清单
- 输出一个简洁的“状态看板字段设计”（Todo/Doing/Blocked/Done）
```

## Prompt 5：Demo 与验收 SOP 对齐

**推荐模型**
- 首选：`Sonnet 4.6`
- 低成本备选：`Kimi K2.5`
- 代码与逻辑核对：`GPT-5.3 Codex` 或 `Opus 4.6`

**可直接复制**

```text
请联动优化 `weekly-prep-docs/01-product/demo.md` 与 `weekly-prep-docs/03-process/开发计划与验收SOP.md`，保证“演示内容”和“验收标准”完全一致。

目标：
1) Demo 讲解有业务价值主线，不只是功能罗列
2) 每个演示步骤都能映射到验收条目
3) 异常场景（库存不足、地址异常、支付失败）有清晰演示与验收口径

输出要求：
- 先给差异清单（demo 与 SOP 不一致之处）
- 再给修订后的两份文档结构建议
- 最后给一版可直接使用的 15 分钟演示脚本（含时间分配）
- 给评审提问清单和标准回答要点
```

## 通用提问骨架（建议每次都带）

```text
【目标】

涉及哪些文件/模块？当前已知问题是什么？

【约束】
不能改什么？技术栈限制是什么？时间限制是什么？
```

## 开发使用模型表（按当前可用模型 + Token 成本）

> 说明：以下成本为相对成本分级（低/中/高），用于做日常选择，不代表官方精确计费单价。

| 模型 | 适用场景 | 能力强项 | 相对 Token 成本 | 默认策略 |
|---|---|---|---|---|
| `Kimi K2.5` | 日常开发、文档整理、需求拆解 | 速度和质量平衡 | 低-中 | 作为默认模型 |
| `Grok Code` | 方案发散、快速草稿、备选思路 | 创意和快速输出 | 低-中 | 先发散，再切主力收敛 |
| `GPT-5.3 Codex` | 代码实现、重构、修复常规 Bug | 代码落地稳定 | 中 | 代码任务首选之一 |
| `GPT-5.4` | 复杂逻辑、跨文件设计、疑难问题 | 深度推理更强 | 中-高 | 卡点升级使用 |
| `Sonnet 4.6` | 代码+文档双栈、沟通表达 | 输出结构清晰 | 中 | 日常主力可替代 Kimi |
| `Opus 4.6` | 架构评审、疑难故障、深度分析 | 复杂问题上限高 | 高 | 仅用于高价值任务 |
| `Composer 1.5` | 多文件连续任务、计划到执行 | 流程编排能力强 | 中-高 | 长任务优先 |
| `Composer 1` | 轻量流程任务、兼容备份 | 稳定兜底 | 中 | 1.5 不稳时切换 |
| `GPT-5.3 Codex Low` | 批量低风险改动 | 成本友好 | 低 | 模板化任务优先 |
| `GPT-5.3 Codex Low Fast` | 简单重复改动、速度优先 | 吞吐高 | 低 | 批处理首选 |

### 成本优先的默认路由

- `L1（低成本层）`：`Kimi K2.5` -> `Grok Code` -> `GPT-5.3 Codex Low Fast`
- `L2（平衡层）`：`Sonnet 4.6` -> `GPT-5.3 Codex` -> `Composer 1.5`
- `L3（高质量层）`：`GPT-5.4` -> `Opus 4.6`

### 任务到模型映射（开发实操）

| 任务类型 | 首选 | 备选 | 成本控制建议 |
|---|---|---|---|
| 页面样式优化（HTML/CSS） | `Kimi K2.5` | `Sonnet 4.6` | 单次上下文控制在 2-4 个文件 |
| Vue 组件开发/重构 | `GPT-5.3 Codex` | `Sonnet 4.6` | 先给接口和边界，减少来回对话 |
| 常规 Bug 修复 | `GPT-5.3 Codex` | `Kimi K2.5` | 每次只修 1 类问题 |
| 疑难/偶现 Bug | `GPT-5.4` | `Opus 4.6` | 先让模型只做根因树，再执行修复 |
| 架构文档评审 | `Opus 4.6` | `GPT-5.4` | 拆两轮：先评审后落地，避免一次超长输入 |
| Demo/SOP 文档优化 | `Sonnet 4.6` | `Kimi K2.5` | 先给模板再让模型填充 |
| 多文件联动改造 | `Composer 1.5` | `Composer 1` | 分批执行（每批 3-5 文件） |

### Token 费用控制规则（建议固定执行）

- 单次对话优先控制在 `<= 8k` token，复杂任务分轮次执行。
- 先用低成本层出方案，再用高质量层做最终决策。
- 同一问题连续失败 2 次才升级到更高成本模型。
- 大文档不要整份反复喂入，只给“变更段落 + 目标 + 约束”。
- 多文件任务采用“计划 -> 执行 -> 验收”三段式，避免一次性超长上下文。

### 推荐默认配置（你当前环境）

- 默认模型：`Kimi K2.5`
- 代码主力：`GPT-5.3 Codex`
- 文档主力：`Sonnet 4.6`
- 深度分析：`Opus 4.6`
- 流程编排：`Composer 1.5`
- 低成本批处理：`GPT-5.3 Codex Low Fast`
