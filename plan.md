# 买菜 App（每周快速备菜）方案文档

## 一、可行性结论

- 方向可行，且需求高频：每周都要买菜、做饭，具备稳定使用场景。
- 差异化空间明确：不做“泛买菜电商”，聚焦“每周备菜决策 + 清单执行”。
- MVP 可快速落地：先做计划、清单、库存、执行闭环，不依赖自建履约。
- 核心风险可控：主要在内容冷启动与留存，通过模板化和周复用机制缓解。

---

## 二、产品定位

### 1）目标用户
- 25-40 岁上班族 / 小家庭
- 每周做饭 3 次以上
- 关注省时间、省钱、减少浪费

### 2）核心痛点
- 不知道一周吃什么，决策成本高
- 买菜容易买漏、买多、买错
- 工作日没时间，临时补货频繁

### 3）一句话价值
**10 分钟完成一周菜单和购物清单，平日 15-30 分钟快速做饭。**

---

## 三、PRD（V1，对齐 demo v0.2 与 bugFix v1）

### 1）产品目标
围绕“每周备菜”建立闭环：
1. 周计划生成  
2. 自动清单整理  
3. 库存扣减 + 执行追踪  

### 2）核心功能（V1）
- 周菜单规划（人数、天数、口味、预算）
- 菜单生成来源（模板库 + 手动替换，不做智能推荐）
- 首版提供 30~50 道高频家常菜模板，覆盖常见午/晚餐场景
- 自动购物清单（别名归一 + 单位换算 + 同类合并 + 按品类排序）
- `pieceToGramMap` 首版覆盖 30~50 个高频食材，超出范围提示手动确认单位
- 库存管理（已有食材扣减、可回滚、幂等）
- 备菜指引（周末预处理 + 工作日快做）
- 模板复用（下周一键复制，白名单复制 + 状态重置）
- 数据备份（JSON 导出与导入恢复，仅支持全量覆盖）
- 导入恢复流程：覆盖前自动导出当前快照，覆盖后提示恢复成功与恢复时间

### 3）V1 不做
- 自营配送
- 社区/短视频
- 复杂营养体系
- 多人协作与聊天
- 自动价格抓取与复杂预算预测

### 4）主流程
首页 -> 新建计划 -> 生成菜单 -> 生成清单 -> 库存扣减 -> 采购 -> 备菜执行 -> 周复盘

### 5）成功指标（V1）
- 计划创建转化率 >= 60%
- 清单生成率 >= 80%
- 周计划完成率 >= 35%
- 次周留存 >= 25%
- 数据恢复成功率 = 100%（导入后核心数据可用）

## 四、用户访谈提纲（20 问）

### A. 行为现状
1. 每周做饭几次？谁负责买菜？

3. 每周预算大约多少？  
4. 一次买一周，还是分次买？  

### B. 痛点挖掘
5. 决定“吃什么”平均花多久？  
6. 最常见问题是买多、买漏还是买错？  
7. 你最烦买菜流程的哪一步？  
8. 最常浪费的食材是什么？为什么？  

### C. 当前替代方案
9. 现在怎么做菜单计划？  
10. 用过清单类 App 吗？为什么没持续？  
11. 有固定家常菜单吗？  
12. 有提前备菜习惯吗？  

### D. 价值验证
13. 若 10 分钟生成一周菜单，愿意用吗？  
14. 自动清单 + 库存扣减价值打分（1-10）？  
15. 更看重省时间、省钱还是营养均衡？  
16. 希望收到哪些提醒（计划/采购/解冻/烹饪）？  

### E. 付费与留存
17. 哪些功能会让你愿意付费？  
18. 可接受月费（9/19/29 元）？  
19. 什么情况下会卸载？  
20. 连续使用 4 周后，你最看重哪个结果？

## 五、MVP 优先级（MoSCoW）

### Must（必须）
- 周计划创建
- 菜单生成与替换
- 清单自动合并（含别名/单位规则）
- 清单支持手动新增食材（同名同单位自动合并）
- 库存扣减（含流水与回滚）
- 采购三态状态（待购买/已购买/冰箱已有，互斥）
- 周计划保存/复用（白名单复制）
- JSON 导出/导入

### Should（应该）
- 菜谱耗时标签（15/30/45 分钟）
- 备菜步骤引导
- 预算对比提示（budget vs actualCost）
- 周提醒

### 数据口径（V1）
- `actualCost` 记入周复盘数据，与 `budget` 做对比并给出超预算高亮提示。

### Could（可选）
- 家庭共享
- 营养标签
- 语音录入库存
- 清单导出图片

### Won’t（本期不做）
- 自营履约配送
- 社区生态
- 复杂营养报告
- 商城体系

## 八、风险与应对

### 1）留存风险
- 风险：首周使用后流失  
- 应对：模板复用、自动提醒、周复盘反馈（省时/省钱）

### 2）内容质量风险
- 风险：菜谱与食材数量不准  
- 应对：先上线 30~50 道高频家常菜模板，并通过手动替换保证可用性

### 2.1）数据一致性风险
- 风险：库存重复扣减、反勾选不回滚  
- 应对：新增 `InventoryLog`，所有库存变更走流水，按 `sourceId` 幂等

### 2.2）清单聚合准确性风险
- 风险：别名、单位不统一导致合并错误  
- 应对：上线 `ingredientAliasMap`、`unitConvertMap`、`pieceToGramMap`

### 3）履约依赖风险
- 风险：接入配送体系复杂  
- 应对：先做清单工具，后期对接第三方跳转下单

### 4）范围失控风险
- 风险：功能过多导致延期  
- 应对：严格控制 MVP 边界，不做社区和重运营模块

### 5）数据丢失风险
- 风险：单用户本地使用中数据意外丢失  
- 应对：IndexedDB 作为主存储 + 每周 JSON 导出备份 + 导入前自动快照

## 九、3 个月推进计划

### 第 1 月：需求验证
- 访谈 20 位目标用户
- 原型验证主流程
- 目标：强意愿用户比例 > 40%

### 第 2 月：MVP 上线
- 发布计划+清单+库存功能
- 目标：清单生成率 > 80%，次周留存 > 20%

### 第 3 月：留存优化
- 强化模板复用、提醒、复盘
- 目标：周计划完成率 > 35%，4 周留存提升

## 十、立即执行建议

1. 先完成规则底座：食材别名归一、单位换算、个转 g 映射  
2. 实现库存流水与回滚，保证“可追踪、可恢复、可幂等”  
3. 落地周复制白名单，禁止继承采购/库存执行状态  
4. 接入 IndexedDB + JSON 导出/导入，建立最小备份闭环  
5. 记录核心数据：决策耗时、采购耗时、漏买次数、浪费情况  
6. 后端方案统一：阶段 A 采用本地模式；阶段 B 固定 `Fastify + JavaScript`，以 `SQLite` 起步，后续按多端同步需求升级 `PostgreSQL`

## 十二、关键节点验证与每周验收 Checklist

### 1）关键节点验证办法（里程碑）

| 里程碑 | 验证动作 | 通过标准 | 失败处理 |
|---|---|---|---|
| M1 规则引擎正确性（W1D5） | 用别名+多单位+“个”混合样例生成清单 | 执行顺序固定（别名->换算->个转 g->聚合），结果一致 | 标记 `needsManualConfirm`，禁止静默合并 |
| M2 库存幂等回滚（W2D2） | 同一清单项连续勾选/取消 5 次 | 无重复扣减；取消后库存可回滚；日志前后值一致 | 锁定该项并提示异常，进入缺陷修复 |
| M3 周复制规则（W2D3） | 复制上周到新周并逐字段比对 | 白名单字段复制；执行态全部重置 | 复制失败不落库，保持旧数据不变 |
| M4 导入恢复可靠性（W2D4） | 导出->清空->导入->全量比对 | 导入前快照存在；导入后核心数据可用 | 自动提示回退到快照 |
| M5 主链路验收（W2D6） | 跑完整链路 E2E | 创建->菜单->清单->库存->采购->复制 全通过 | 阻断发布，先清零 P0 |
| M6 4 周价值验证（W6） | 汇总每周指标并对照门槛 | 满足 Go 门槛 | 不满足则 No-Go，暂停扩展投入 |

- [ ] 清单支持手动新增食材并正确归类

- [ ] 库存勾选可扣减并写入 `InventoryLog`
- [ ] 反勾选可回滚库存且无重复扣减
- [ ] 可勾选采购状态并保留记录
- [ ] 可复制上周计划且执行状态已重置
- [ ] 可导出 JSON 并导入恢复核心数据

#### B. 质量门禁
- [ ] P0 缺陷 = 0（库存/清单/导入恢复相关）
